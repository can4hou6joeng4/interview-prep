支付场景下的幂等怎么设计？Checkout ID、Trade ID 和 Webhook 去重分别怎么保证？

困难 projectscene

三层幂等机制

  • Checkout 层：同一个 Checkout ID 只允许创建一笔成功的 Payment，重复提交直接返回已有结果
  • Trade 层：每次支付尝试生成唯一 Trade ID 传给支付网关（PayPal/Stripe），网关侧保证同一 Trade ID 不会重复扣款
  • Webhook 层：支付网关可能重复推送回调通知，用 webhook_event_id 做去重，已处理过的事件直接返回 200 不重复执行

为什么需要三层而不是一层

  • Checkout 层防的是用户重复点击提交按钮
  • Trade 层防的是网络超时重试导致网关侧重复扣款
  • Webhook 层防的是网关重复推送导致本地重复处理订单状态

实现细节

// Webhook 去重伪代码
func HandleWebhook(eventId string, payload []byte) error {
    // 1. 检查是否已处理
    if exists := redis.SetNX("webhook:"+eventId, 1, 24*time.Hour); !exists {
        return nil // 已处理，直接返回成功
    }
    // 2. 处理业务逻辑
    err := processPayment(payload)
    if err != nil {
        redis.Del("webhook:" + eventId) // 处理失败，允许重试
        return err
    }
    return nil
}
面试核心：幂等不是"加个唯一索引"就完事，而是要在每一层分别防御不同来源的重复请求。
