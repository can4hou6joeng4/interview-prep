Capability 套餐能力控制系统是怎么设计的？为什么用 Assert 模式？

困难 projectscene

三层模型

  • Plan（套餐）：Free / Pro / Enterprise，定义在配置中，运行时通过 planCatalogSource() 加载
  • Quota（配额）：每个套餐定义资源上限，如 ProductMax、CustomDomainMax、ThemeInstallMax、EmployeeMax，-1 表示无限制
  • Feature（功能开关）：布尔开关控制高级功能是否可用，如 RecoveryEnabled（营销召回）

什么是 Assert 模式

Assert 模式的语义类似断言："断言当前操作在配额范围内，否则中断"。在需要校验配额的业务代码点，主动调用 Assert 方法拦截。

// 创建商品前校验商品配额
func (s *ProductService) Create(ctx context.Context, input ProductInput) error {
    if err := capability.AssertProductLimit(ctx, s.db, 1); err != nil {
        return err // 超出套餐商品数上限，拦截创建
    }
    // ... 正常创建逻辑
}

// Assert 内部实现
func AssertProductLimit(ctx context.Context, db *gorm.DB, incoming int) error {
    plan := loadStorePlan(ctx, db)       // 加载当前店铺套餐快照
    used := countProducts(ctx, db)       // 查询实际已用数量
    limit := plan.Quota.ProductMax       // 获取配额上限
    if limit != -1 && used+incoming > limit {
        log.Warn("product limit exceeded", "source", "product.create", "used", used, "limit", limit)
        return ErrQuotaExceeded
    }
    return nil
}

为什么不用统一中间件拦截

  • 校验时机不同：创建商品、绑定域名、触发召回任务发生在完全不同的业务入口，统一中间件无法知道当前请求涉及哪种资源操作
  • 调用点明确：Assert 嵌入具体业务代码中，每个方法都带 source 参数（如 product.create、domain.bindCustom），日志里一眼就能追踪是哪个入口触发的
  • 精确校验：中间件只能做"有没有权限"的粗粒度判断，Assert 可以做"当前余量够不够"的增量校验

Assert 模式 vs 常见替代方案

方案适用场景不适用场景
Assert 模式（本项目）资源配额校验、增量判断、校验时机分散统一的角色权限控制
RBAC 中间件角色权限校验（管理员/普通用户）资源余量判断（它只知道角色，不知道数量）
装饰器/AOP统一的横切关注点（日志、鉴权）每个校验点逻辑不同的场景

套餐降级后超限怎么处理

  • 采用"存量不删、增量拦截"策略：降级后已有的商品、域名、模板不会被自动删除（避免数据丢失），但新增操作会被 Assert 拦截
  • 前端通过 Current() 接口获取实时配额信息（已用/剩余/上限），展示升级引导
简历关联：Shoply 的套餐系统采用 Plan + Quota + Feature 三层模型，并通过 Assert 模式做增量拦截。
