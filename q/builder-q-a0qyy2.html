积分系统为什么用 Builder 模式？事务内行锁怎么保证余额一致性？

困难 project

为什么用 Builder 模式

积分变更涉及多个参数组合（credits/points/ruleCode/ruleId/userId/remark），不同业务场景需要传入的参数不同。Builder 模式的好处：

NewCredit(db).
    WithCredits(10).
    WithPoints(50).
    WithRuleCode("order_complete").
    WithUserId(uid).
    Do()
  • 可读性：链式调用语义清晰，一眼看出这次操作做了什么
  • 灵活性：只传需要的参数，不用构造一个大 struct 填一堆零值
  • 校验内聚：Do() 内部统一校验必填参数，外部无需关心校验逻辑

事务内行锁保证一致性

func (b *CreditBuilder) Do() error {
    return b.db.Transaction(func(tx *gorm.DB) error {
        // 1. SELECT FOR UPDATE 锁定用户行
        var user User
        tx.Clauses(clause.Locking{Strength: "UPDATE"}).
            First(&user, b.userId)
        // 2. 幂等校验：sourceId 防重复发放
        if exists(tx, b.sourceId) { return nil }
        // 3. 计算新余额，校验不能透支
        newBalance := user.Credits + b.credits
        if newBalance < 0 { return ErrInsufficientCredits }
        // 4. 更新余额 + 写流水日志
        tx.Model(&user).Update("credits", newBalance)
        tx.Create(&CreditLog{...})
        return nil
    })
}

关键设计点

  • 行锁：SELECT FOR UPDATE 锁住当前用户行，防止并发修改余额不一致
  • 幂等：sourceId 唯一校验，同一笔订单不会重复发放积分
  • 事务包裹：余额更新和流水日志在同一事务中，失败自动回滚
