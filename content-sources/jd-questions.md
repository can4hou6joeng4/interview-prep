## 三、需要新增的面试题（按 JD 缺口分析）

### 3.1 gRPC 与 Protobuf（欢聚、微派、道云要求）

- **gRPC 的四种通信模式是什么？（Unary / Server Stream / Client Stream / Bidirectional Stream）**
  - 类型：基础
  - 关联岗位：欢聚集团、道云
  - 答案要点：Unary 一问一答；Server Stream 服务端推流（如实时价格推送）；Client Stream 客户端推流（如文件上传）；Bidirectional 双向流（如聊天）

- **Protobuf 的编码原理？为什么比 JSON 快？**
  - 类型：基础
  - 答案要点：Varint 变长编码 + Tag-Length-Value 格式；二进制序列化（vs JSON 文本）；无字段名（用编号替代）；体积约 JSON 的 1/3，速度快 3-10 倍

- **gRPC 的拦截器 (Interceptor) 怎么用？和 HTTP 中间件有什么类似？**
  - 类型：进阶
  - 关联岗位：微派（服务治理）
  - 答案要点：UnaryInterceptor / StreamInterceptor；链式调用；常用于认证、日志、限流、链路追踪

- **gRPC 的负载均衡方案？客户端负载 vs 代理负载？**
  - 类型：进阶
  - 答案要点：客户端 LB（grpc-go 内置 round_robin/pick_first）；代理 LB（Envoy/Nginx）；Name Resolver + Balancer 架构

### 3.2 性能优化实战（微派、舜飞、智品强调）

- **Go 程序的性能分析工具链有哪些？pprof / trace / benchmark 分别解决什么问题？**
  - 类型：进阶
  - 关联岗位：微派（明确要求 pprof/火焰图）
  - 答案要点：`pprof` CPU/内存/goroutine/block profiling；`trace` 可视化 goroutine 调度、GC、系统调用时间线；`benchmark` 函数级性能基准测试（`go test -bench`）

- **Go 中减少锁竞争的方案？无锁设计思路？**
  - 类型：深入
  - 关联岗位：微派（"锁竞争定位、减锁/无锁设计"）
  - 答案要点：分片锁（sharded mutex）；atomic 原子操作；channel 替代 mutex；sync.Map（读多写少）；Copy-on-Write 模式；lock-free queue（CAS 实现）

- **Redis 热点 Key 和大 Key 怎么治理？**
  - 类型：进阶
  - 关联岗位：微派（"热点与大 Key 治理"）
  - 答案要点：热点 Key — 本地缓存兜底 / key 分散（加后缀）/ 读写分离；大 Key — `MEMORY USAGE` 检测 / 拆分为多个小 Key / 异步删除（UNLINK）/ Hash 分片

- **MySQL 慢查询定位和优化的完整流程？**
  - 类型：进阶
  - 关联岗位：微派（"治理慢查询"）、道云（"优化SQL查询"）
  - 答案要点：开启 slow_query_log → pt-query-digest 分析 → EXPLAIN 执行计划 → 索引优化 / 查询重写 / 分表 → 验证效果

- **线上接口响应变慢，排查思路是什么？**
  - 类型：深入（场景题）
  - 关联岗位：多岗位
  - 答案要点：看监控（P99/P95） → 看日志 → pprof CPU/goroutine → trace 看调度 → 慢查询日志 → Redis slow log → 网络抓包 → 分层定位（网关→应用→缓存→DB）

### 3.3 弱网与网络协议优化（微派明确要求）

- **TCP 的拥塞控制算法？慢启动、拥塞避免、快重传、快恢复？**
  - 类型：进阶
  - 关联岗位：微派（"拥塞控制、背压"）
  - 答案要点：慢启动（cwnd 指数增长）→ 拥塞避免（cwnd 线性增长）→ 快重传（3 个重复 ACK 触发重传）→ 快恢复（cwnd 减半而非置 1）

- **弱网场景下的优化策略？超时重试、断线重连、背压？**
  - 类型：深入（场景题）
  - 关联岗位：微派（明确要求 "弱网优化意识"）
  - 答案要点：指数退避重试 + jitter；心跳探活 + 断线重连（WebSocket）；请求幂等保证；客户端本地队列 + 批量上传；服务端背压（channel 满时丢弃/限流）；TCP KeepAlive 调优

- **HTTP/2 的 Stream 多路复用在 gRPC 中如何体现？**
  - 类型：进阶
  - 关联岗位：微派（要求了解 HTTP/2、QUIC/HTTP/3）
  - 答案要点：gRPC 基于 HTTP/2，一个 TCP 连接上并行多个 RPC 调用（每个是一个 Stream）；帧（Frame）交错传输；HPACK 头部压缩减少带宽

### 3.4 AI 应用与 LLM 工程（网易、腾讯、微派、道云要求）

- **LangChain 的核心概念？Chain / Agent / Tool / Memory 分别是什么？**
  - 类型：基础
  - 关联岗位：道云（LangChain 工程师）
  - 答案要点：Chain 串联 LLM 调用链；Agent 根据输入动态选择 Tool 执行；Tool 外部能力（搜索/计算/API）；Memory 对话历史管理（Buffer/Summary/Vector）

- **如何设计一个 AI Agent 的后端架构？任务编排、上下文管理、错误处理？**
  - 类型：深入（场景题）
  - 关联岗位：道云、网易（AI 应用）
  - 答案要点：任务队列（Asynq/Celery）+ LLM API 调用 + 工具执行 + 上下文窗口管理（Token 裁剪/摘要）+ 结构化输出（JSON Schema）+ 重试与 fallback + 流式输出（SSE/WebSocket）
  - 简历关联：你有 AI Agent 后端落地经验

- **Prompt Engineering 的基本技巧？Few-shot / Chain-of-Thought / ReAct？**
  - 类型：基础
  - 关联岗位：道云、网易
  - 答案要点：Zero-shot（直接指令）；Few-shot（提供示例）；CoT（引导推理过程）；ReAct（Reasoning + Acting，Agent 常用）；System Prompt 设计原则

- **LLM 应用中的 RAG (Retrieval-Augmented Generation) 架构？**
  - 类型：进阶
  - 关联岗位：道云
  - 答案要点：文档分块 → Embedding → 向量数据库存储 → 用户 Query Embedding → 相似度检索 → 拼接 Context → LLM 生成答案；常见向量库：Milvus/Weaviate/Pinecone/pgvector

### 3.5 程序化广告系统（舜飞科技要求）

- **程序化广告的 RTB（实时竞价）流程？DSP/SSP/ADX 的角色？**
  - 类型：基础
  - 关联岗位：舜飞科技（程序化广告、ADX、SSP）
  - 答案要点：用户访问页面 → SSP（供给方平台）发出广告请求 → ADX（广告交易平台）向多个 DSP（需求方平台）发起竞价 → DSP 返回出价 → ADX 选最高出价 → 返回广告素材 → 用户看到广告。全流程需在 100ms 内完成

- **高并发广告系统的技术挑战？QPS 要求和延迟要求？**
  - 类型：进阶（场景题）
  - 关联岗位：舜飞科技（高并发、性能优化）
  - 答案要点：QPS 通常 10万+；延迟 <100ms；Go 天然适合（goroutine 高并发）；预计算 + 内存缓存（频控/预算）；异步日志（不阻塞主链路）；数据库写入批量异步化

### 3.6 跨境电商场景题（欢聚集团要求）

- **跨境电商独立站的核心技术挑战有哪些？**
  - 类型：进阶（场景题）
  - 关联岗位：欢聚集团（跨境电商独立站）
  - 答案要点：多币种支付（与你项目高度匹配）；多语言国际化；时区处理；跨境物流追踪；合规（GDPR/税收）；CDN 全球加速；反欺诈风控
  - 简历关联：你的 Shoply 项目覆盖了多币种、多支付网关、国际化等所有核心点

### 3.7 服务可观测性（微派、道云要求）

- **可观测性三支柱是什么？Metrics / Logs / Traces 的区别？**
  - 类型：基础
  - 关联岗位：微派（"观测性建设"）、道云（"监控"）
  - 答案要点：Metrics — 数值指标（Prometheus + Grafana）；Logs — 事件记录（ELK / Loki）；Traces — 请求链路（Jaeger / OpenTelemetry）；三者结合才能快速定位问题

- **SLO/SLI/SLA 是什么？如何定义和监控？**
  - 类型：进阶
  - 关联岗位：微派（"SLO"）
  - 答案要点：SLI（指标，如 P99 延迟、可用性百分比）；SLO（目标，如 99.9% 可用性）；SLA（协议，对外承诺）；Error Budget = 1 - SLO，用于权衡功能迭代速度与稳定性

- **Prometheus + Grafana 监控体系怎么搭建？Go 程序如何暴露指标？**
  - 类型：进阶
  - 关联岗位：微派、道云
  - 答案要点：引入 `prometheus/client_golang`；暴露 `/metrics` 端点；自定义 Counter/Gauge/Histogram/Summary；Grafana 配置 Dashboard + 告警规则

### 3.8 CI/CD 与 DevOps（网易、道云要求）

- **Go 项目的 CI/CD 流程一般怎么设计？**
  - 类型：基础
  - 关联岗位：网易（DevOps）、道云（CI/CD 流程建设）
  - 答案要点：代码推送 → lint (golangci-lint) → 单元测试 (go test) → 构建 (go build) → Docker 镜像 → 推送镜像仓库 → K8s 滚动更新；常用工具：GitHub Actions / GitLab CI / Jenkins

- **灰度发布/金丝雀发布怎么实现？**
  - 类型：进阶
  - 关联岗位：微派（"灰度/回滚"）
  - 答案要点：K8s 层面 — 两个 Deployment 不同版本，通过 Service/Ingress 按比例分流；Istio — VirtualService 权重路由；应用层 — Feature Flag + 用户 ID 哈希分桶；监控指标异常自动回滚

### 3.9 算法与数据结构手撕补充（腾讯、网易、微派要求）

- **实现一个 Go 版本的跳表 (Skip List)，理解 Redis ZSet 底层**
  - 类型：深入
  - 答案要点：多层链表，每层跳跃不同步长；查找/插入/删除均为 O(log n)；Redis ZSet 在 128+ 元素时使用 skiplist

- **实现 Top K 问题（海量数据中找前 K 大的元素）**
  - 类型：进阶
  - 关联岗位：腾讯、网易（算法功底）
  - 答案要点：小顶堆（min heap）维护 K 个元素；遍历数据，大于堆顶则替换；时间 O(n log k)，空间 O(k)

- **实现一个简单的布隆过滤器 (Bloom Filter)**
  - 类型：进阶
  - 答案要点：位数组 + 多个哈希函数；判断元素"可能存在"或"一定不存在"；用于缓存穿透防护、URL 去重

- **用 Go 实现生产者-消费者模型（channel + select）**
  - 类型：基础
  - 答案要点：buffered channel 做队列；多 producer goroutine 写入；多 consumer goroutine 读取；select + context 优雅退出

## 七、第二批新增面试题

### 7.1 DDD 领域驱动设计（AfterShip 明确要求）

- **DDD 的核心概念？实体、值对象、聚合根、领域事件分别是什么？**
  - 类型：进阶
  - 关联岗位：AfterShip（"DDD 深度实践经验优先"）
  - 答案要点：Entity（有唯一标识，如 Order）；Value Object（无标识，如 Money/Address）；Aggregate Root（聚合根，事务一致性边界，如 Order 管理 OrderItem）；Domain Event（领域事件，解耦上下游，如 OrderCreated）；Bounded Context（限界上下文，微服务拆分依据）

- **DDD 分层架构？和传统 MVC 三层架构有什么区别？**
  - 类型：进阶
  - 答案要点：DDD 四层 — Interface（接口层/API）→ Application（应用层/用例编排）→ Domain（领域层/核心业务逻辑）→ Infrastructure（基础设施/DB/MQ）。与 MVC 区别：领域逻辑从 Service 层下沉到 Domain 层，Repository 接口在 Domain 层定义、Infrastructure 层实现（依赖倒置）

- **Go 项目如何实践 DDD？目录结构怎么组织？**
  - 类型：深入（场景题）
  - 关联岗位：AfterShip
  - 答案要点：按限界上下文划分模块（`internal/order/`、`internal/payment/`）；每个模块内 `domain/`（实体+仓储接口）、`application/`（用例）、`infrastructure/`（实现）、`interfaces/`（HTTP/gRPC handler）；Go 中用 interface 实现依赖倒置

### 7.2 K8s Operator 开发（查特兰海明确要求）

- **什么是 K8s CRD 和 Operator？Kubebuilder 的工作原理？**
  - 类型：深入
  - 关联岗位：查特兰海（"CRD/Controller/Operator 开发"）
  - 答案要点：CRD（Custom Resource Definition）让用户定义自定义资源类型（如 MySQLCluster）；Operator = CRD + Controller，用代码自动化管理自定义资源的全生命周期；Kubebuilder 是 Operator 脚手架，基于 controller-runtime，生成 Reconciler 框架代码；核心循环：Watch 资源变化 → Reconcile（对比期望状态与实际状态）→ 执行操作（创建/更新/删除 K8s 对象）

- **Operator 的 Reconcile 循环是什么？如何保证幂等性？**
  - 类型：深入
  - 答案要点：Reconcile 函数被反复调用（事件驱动 + 定期 resync）；必须幂等 — 每次执行结果相同；使用 `CreateOrUpdate` 而非先判断再创建；用 Status subresource 记录状态，避免无限循环；设置 ownerReference 自动级联删除

### 7.3 搜广推系统基础（滴滴要求）

- **搜索/推荐/广告系统的核心架构？召回→粗排→精排→重排的流程？**
  - 类型：进阶
  - 关联岗位：滴滴（"搜广推"）
  - 答案要点：召回（从亿级候选集缩小到千级，多路召回：协同过滤/向量检索/规则）→ 粗排（轻量模型快速打分，千→百）→ 精排（复杂模型精确排序，百→十）→ 重排（业务策略：去重/多样性/广告插入）；后端负责的部分：特征服务（实时特征计算+缓存）、在线推理服务、A/B 实验平台、日志埋点

- **特征工程中实时特征和离线特征的区别？Feature Store 是什么？**
  - 类型：进阶
  - 答案要点：离线特征 — 批量计算（Spark/Hive），更新频率小时/天级，如用户历史行为统计；实时特征 — 流式计算（Flink/Kafka Streams），秒级更新，如最近 5 分钟点击数；Feature Store — 统一管理特征的平台，解决特征一致性（训练/推理用同一份特征）和复用问题

### 7.4 AWS 云服务实践（计算美学要求）

- **AWS 核心服务有哪些？EC2/S3/RDS/Lambda 分别适用什么场景？**
  - 类型：基础
  - 关联岗位：计算美学（AWS 方向）
  - 答案要点：EC2（虚拟机，通用计算）；S3（对象存储，静态资源/备份/数据湖）；RDS（托管数据库，MySQL/PostgreSQL）；Lambda（Serverless 函数，事件驱动短任务）；ECS/EKS（容器编排）；SQS（消息队列）；CloudFront（CDN）
  - 简历关联：你的 go-storage 支持 OSS 对接，切到 AWS S3 只需新增一个 S3Driver

- **如何设计一个可靠的 AWS 部署架构？高可用和容灾怎么做？**
  - 类型：进阶（场景题）
  - 答案要点：多 AZ 部署（跨可用区）；ALB + Auto Scaling Group（弹性扩缩）；RDS Multi-AZ（主备切换）；S3 跨区域复制；CloudWatch 监控 + SNS 告警；Route 53 故障转移路由

### 7.5 AIGC 业务架构（回响科技、AfterShip、查特兰海共同要求）

- **AI 大模型推理部署方案？vLLM / TGI / Triton 的区别？**
  - 类型：深入
  - 关联岗位：查特兰海（明确要求 vLLM/KServe）
  - 答案要点：vLLM — 高性能 LLM 推理引擎，PagedAttention 优化显存，吞吐量最高；TGI（Text Generation Inference）— HuggingFace 出品，易用性好；Triton — NVIDIA 通用推理服务器，支持多框架（PyTorch/TensorFlow/ONNX）；KServe — K8s 上的 AI 模型服务框架，自动扩缩容 + 金丝雀发布

- **GPU 调度在 K8s 中怎么实现？资源管理有什么挑战？**
  - 类型：深入
  - 关联岗位：查特兰海（"GPU 调度"）
  - 答案要点：NVIDIA Device Plugin 暴露 GPU 资源（`nvidia.com/gpu`）；Pod 通过 `resources.limits` 申请 GPU；挑战：GPU 不能像 CPU 一样细粒度分时共享（MIG/MPS 方案）；GPU 利用率监控（DCGM Exporter + Prometheus）；调度策略：bin-packing（填满一个节点再用下一个）减少碎片

- **Dify / LangChain 等 AI 应用开发框架的区别？**
  - 类型：基础
  - 关联岗位：腾音（"LangChain、Dify"）
  - 答案要点：LangChain — 代码级 AI 应用框架（Python/JS），灵活但学习曲线高；Dify — 可视化 AI 应用开发平台，低代码拖拽编排 workflow，适合快速搭建；两者关系类似 Spring Boot vs 低代码平台

### 7.6 单元测试与代码质量（AfterShip 强调）

- **Go 项目如何做好单元测试？Table-Driven Tests / Mock / 测试覆盖率？**
  - 类型：进阶
  - 关联岗位：AfterShip（"单元测试深度实践"）
  - 答案要点：Table-Driven Tests（数据驱动，一个函数多组用例）；接口 Mock（gomock / testify/mock）；httptest 测 HTTP Handler；testcontainers-go 集成测试（真实 DB/Redis）；覆盖率 `go test -cover`，CI 中设置最低覆盖率门槛
  - 简历关联：你的项目有完整的测试策略（test_auth / test_api / test_commands 等）

- **TDD（测试驱动开发）的流程？Red-Green-Refactor 是什么？**
  - 类型：基础
  - 答案要点：Red — 先写一个会失败的测试；Green — 写最少的代码让测试通过；Refactor — 重构代码保持测试绿色；好处：设计更好的 API、防止过度设计、回归安全网

### 12.2 跨境电商交易深化（欢聚集团重点）

- **Checkout 结算引擎为什么适合用 Pipeline 模式？如何保证促销、优惠券、运费、税费计算的扩展性和顺序正确？**
  - 类型：深入（场景题）
  - 关联岗位：欢聚集团（跨境电商独立站）
  - 答案要点：把 `prepareBaseItems → applyPromotions → applyCoupons → applyExternalDiscounts → applyShipping → applyTax → summarize → applyCurrency` 拆成独立 calculator；每一步只处理单一职责；通过上下文对象传递中间结果；顺序由业务语义而不是代码先后偶然决定；支持新增步骤但不破坏既有流程
  - 简历关联：你的 TradeCalc 结算 Pipeline 正好可以作为完整案例

- **优惠、运费、税费按商品行分摊时，为什么容易出错？如何处理精度、尾差和退款反算？**
  - 类型：深入
  - 关联岗位：欢聚集团
  - 答案要点：不能用 float，要用 decimal；分摊基于权重（金额、数量、重量等）计算；尾差通过“最大项补差”或“最后一项补差”收敛；把分摊结果落快照，退款按原始分摊回放，不能重新按当前规则计算

- **为什么下单链路要做商品快照、地址快照、汇率快照？哪些字段必须冻结？**
  - 类型：进阶
  - 关联岗位：欢聚集团
  - 答案要点：避免商品改价、标题变更、地址变更、汇率波动影响已下单订单；至少冻结 SKU、标题、规格、单价、优惠、税区、运费规则命中结果、汇率、收货地址关键信息；快照是支付、退款、售后、审计的依据

- **运输费和税费规则引擎怎么设计？国家/省份匹配、价内税/价外税、外部物流报价如何统一抽象？**
  - 类型：深入（场景题）
  - 关联岗位：欢聚集团
  - 答案要点：规则匹配采用“省份 → 国家 → 默认”降级；运费支持内部报价表和外部物流 API 双模式；税费同时支持 VAT 价内税和 Sales Tax 价外税；折扣是否税前/税后生效要可配置；命中的规则和报价结果必须可审计

### 12.3 平台 / 基础设施 / 框架型岗位（网易、兴趣岛、智品重点）

- **平台型基础设施系统和业务型系统在架构设计上有什么不同？**
  - 类型：进阶
  - 关联岗位：网易、兴趣岛、广州智品网络科技
  - 答案要点：平台型系统更强调稳定抽象、自助能力、审计、权限、扩展性和可运维性；业务型系统更强调快速支撑具体需求；平台系统的数据模型通常更偏元数据与流程，不应把业务逻辑硬编码进平台内核

- **自动化工单系统如何设计状态机、审批流、幂等和审计？**
  - 类型：深入（场景题）
  - 关联岗位：网易
  - 答案要点：状态通常包含 draft / pending / approved / executing / success / failed / cancelled；审批人与执行人分离；工单提交与执行都要有 idempotency key；每次状态变更写审计日志和操作快照；失败后要有补偿或人工接管路径

- **资源管理系统的数据模型怎么设计？资源、机房、节点、标签、配额、工单之间是什么关系？**
  - 类型：深入
  - 关联岗位：网易
  - 答案要点：资源实体和流程实体分开设计；资源模型至少区分机房/区域、节点、服务实例、标签、负责人、配额；工单不直接等于资源本身，而是对资源执行某种变更申请；适合用标签和关系模型而不是把所有维度硬塞进一张表

- **时序数据库适合什么场景？它和 MySQL、Redis、MongoDB 的边界是什么？**
  - 类型：进阶
  - 关联岗位：网易
  - 答案要点：TSDB 适合指标、监控、埋点、资源使用率等按时间写入和查询的场景；擅长高写入、按时间窗口聚合、降采样、保留策略；MySQL 适合事务和关系查询，Redis 适合热点计数和缓存，MongoDB 适合文档灵活结构，不能把所有时间序列都硬塞进 MySQL

- **配置中心的分层覆盖怎么设计？环境、租户、主题、运行时配置冲突时谁优先？**
  - 类型：深入
  - 关联岗位：网易、兴趣岛、广州智品网络科技
  - 答案要点：通常采用 `default < environment < region < tenant < theme < runtime override` 的覆盖顺序；每层都要可追踪来源；支持 schema 校验、灰度发布和快速回滚；热更新要区分“可热更新项”和“必须重启项”

### 12.4 社交 / 广告 / 实时系统（微派、趣享未来、舜飞、智品重点）

- **语音房 / 直播间服务端的核心模块有哪些？房间状态、在线人数、消息投递如何保证一致性？**
  - 类型：深入（场景题）
  - 关联岗位：趣享未来、微派、广州智品网络科技
  - 答案要点：核心模块一般包括房间服务、会话/在线状态、消息服务、风控与禁言、礼物/互动流水；在线人数通常是“缓存实时值 + 异步校准”；加入/离开要幂等；房间状态变更要有 source of truth，不能完全依赖前端上报

- **程序化广告或数据监测类系统的实时统计链路怎么设计？如何做去重、聚合和延迟控制？**
  - 类型：深入
  - 关联岗位：舜飞科技
  - 答案要点：入口日志先写 MQ/Kafka，按事件时间做窗口聚合；去重依赖 requestId / impressionId / 指纹；热点计数可以先落 Redis，再异步汇总到 OLAP/明细库；需要明确统计口径是实时近似值还是离线校准值，避免口径漂移

- **TCP、UDP、WebSocket、HTTP/2 在实时社交产品里怎么选？分别适合什么链路？**
  - 类型：进阶
  - 关联岗位：微派、趣享未来、广州智品网络科技
  - 答案要点：HTTP/2 / gRPC 更适合服务间 RPC；WebSocket 更适合客户端实时信令与状态同步；UDP / QUIC 更适合低延迟音视频媒体链路；TCP 适合强可靠控制消息；选型要看可靠性、时延、顺序性和连接管理成本

- **“高并发”不是泛词：社交或广告系统里最先爆的通常是哪几层？容量评估怎么做？**
  - 类型：深入（场景题）
  - 关联岗位：微派、舜飞科技、广州智品网络科技
  - 答案要点：常见瓶颈先出现在连接层、缓存热点层、数据库/消息堆积层；容量评估不能只看 QPS，还要看并发连接数、消息吞吐、Redis 热点、队列积压、P99 延迟、错误率；容量结论需要结合压测与线上监控双证据

### 12.5 压测与可观测性闭环（腾讯、网易、微派重复要求）

- **压测方案怎么设计才有意义？除了 QPS，还要看哪些指标才能真正定位瓶颈？**
  - 类型：深入
  - 关联岗位：微派、广州智品网络科技、腾讯
  - 答案要点：压测流量要接近真实读写比例与热点分布；关键指标除了 QPS 还包括 P95/P99、错误率、CPU、内存、GC、goroutine、连接池、缓存命中率、队列 lag、慢查询数；压测结果必须能回到具体代码或组件瓶颈

- **从告警触发到回滚止损，如何把 Metrics、日志、Trace、灰度发布串成一个排障闭环？**
  - 类型：深入（场景题）
  - 关联岗位：腾讯、网易、微派
  - 答案要点：先由监控发现异常，再用 TraceID 串联日志和链路追踪定位问题实例或下游依赖；如果问题只出现在灰度版本，要能快速切流或回滚；排障闭环不是“找到报错”而是“定位根因、止损、验证恢复、沉淀复盘”
