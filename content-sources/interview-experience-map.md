## 结论摘要

可以整理并归纳到题库里，但不建议“一股脑全量并入”。

更合适的做法是分两层：

- **正式题库候选**
  - 与当前仓库定位一致的 Go 后端高频题
  - 在多份面经中重复出现
  - 对当前简历项目和面试场景有明显帮助

- **专项素材池**
  - Web3 / Solana / EVM / 钱包 / 密码学等强领域题
  - 当前仓库还没有对应主题分类
  - 适合以后单独做“区块链后端专题”，不宜直接混入当前站点

## 适合正式入库的新增候选

下面这些题来自你给的面经，和当前 Go 后端题库的主题最契合，值得作为下一轮正式题库候选。

### Go 并发与运行时

- **Go 里的 WaitGroup 是怎么工作的？常见误用有哪些？**
  - 来源：跨境支付结算小厂
  - 价值：属于非常常见的真实面试追问，当前正式题库里还没有单独展开

- **select 同时多个 case 就绪时会怎么选？for-select 循环有哪些坑？**
  - 来源：跨境支付结算小厂
  - 价值：比“select 怎么用”更贴近真实追问

- **协程池怎么设计？为什么 goroutine 不能无上限地开？**
  - 来源：西井科技、量化开发初创公司
  - 价值：高频并发工程题，当前题库还缺一题专门回答“协程池”

- **sync.Map 里存的是引用类型时，为什么并发修改字段仍然可能不安全？**
  - 来源：西井科技
  - 价值：非常好的深入题，能区分“知道 sync.Map”和“真正理解并发安全边界”

### MySQL 与数据层

- **InnoDB 和 MyISAM 在 `count(*)` 上为什么表现不同？**
  - 来源：跨境支付结算小厂
  - 价值：真实业务面里很常见，当前站点还没有这个角度

- **按时间分表后，跨分片范围查询和分页怎么做？**
  - 来源：跨境支付结算小厂
  - 价值：当前题库提到了分表，但还缺“跨时间区间查询”的追问展开

- **为什么很多场景下不建议直接写 `select *`？什么情况下又可以接受？**
  - 来源：跨境支付结算小厂
  - 价值：常见数据库工程习惯题，适合做基础到进阶的过渡题

### Redis / MQ / 分布式

- **Redis 宕机时，缓存层怎么降级？哪些数据可能丢，业务该怎么兜底？**
  - 来源：跨境支付结算小厂
  - 价值：比“持久化区别”更偏线上应急，适合当前题库

- **为什么业务消息队列通常不用 Redis List，而会选 Kafka / RabbitMQ / Asynq？**
  - 来源：跨境支付结算小厂、面试题完整汇总
  - 价值：非常贴近技术选型追问

- **多实例同时拉取同一批数据时，怎么避免重复处理？**
  - 来源：面试题完整汇总
  - 价值：和分布式锁、任务领取、幂等消费能形成一个场景组

- **分布式场景下，最终落盘一致性通常怎么保证？**
  - 来源：面试题完整汇总
  - 价值：可以和 TCC / Saga / 本地消息表等现有分布式一致性题呼应

### gRPC / 容器 / 网络

- **gRPC 的底层实现依赖哪些协议和机制？**
  - 来源：西井科技
  - 价值：现在题库有 gRPC 通信模式和拦截器，但缺“底层原理”向问题

- **Docker 和虚拟机的核心区别是什么？**
  - 来源：西井科技
  - 价值：适合作为 Docker 基础区的补充题

- **TCP 重传是怎么触发的？三次握手后常见的断连场景有哪些？**
  - 来源：西井科技
  - 价值：当前网络题库已覆盖三次握手 / 挥手，但“重传 + 断连原因”还可以补一题

### 微服务治理 / 基础设施

- **服务注册发现的作用是什么？如果放到 K8s 里，哪些组件能承担它？**
  - 来源：量化开发初创公司、跨境支付结算小厂
  - 价值：和现有服务发现题能形成“概念题 + 落地题”组合

## 建议只放专项素材池，不直接并入当前站点

以下题目本身有价值，但和当前 Go 后端题库主线不完全一致，建议先留在素材池：

- BIP44 路径
- HD 钱包助记词生成
- ECDSA / EdDSA
- secp256k1 / r1

- 当前站点主线是“Go 后端岗位 + 现有简历项目”
- 这些题会显著改变站点主题边界
- 更合适的方式是以后单开：
  - `blockchain-backend-prep`
  - 或在当前仓库里新增独立专题后再并入
